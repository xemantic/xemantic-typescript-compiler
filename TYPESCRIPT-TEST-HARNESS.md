# TypeScript Test Harness: Baseline Comparison and Parameterized Tests

This document describes how the TypeScript compiler's test harness handles baseline comparison,
formatting, and parameterized test generation. Based on reading the actual TypeScript source code
in `src/testRunner/compilerRunner.ts`, `src/harness/harnessIO.ts`, and `src/harness/typeWriter.ts`.

---

## Section 1: Baseline Types

The compiler test harness produces six distinct baseline file types for each test case. Each test
case (e.g., `foo.ts`) can generate up to six files in `tests/baselines/reference/`:

### 1.1 `.js` — JavaScript Output Baseline

**Contains:** The echoed TypeScript source followed by emitted JavaScript (and optionally `.d.ts`
declaration output).

**Generated by:** `Compiler.doJsEmitBaseline()` in `harnessIO.ts`.

**Structure:**
```
//// [tests/cases/compiler/foo.ts] ////

//// [foo.ts]
<echoed TypeScript source>

//// [foo.js]
<emitted JavaScript>

//// [foo.d.ts]         ← only if declaration: true
<emitted declarations>
```

For multi-file tests, each source file is echoed separately under its own `//// [filename.ts]`
header, followed by each output file under `//// [filename.js]` headers.

**Special cases:**
- If JS output has parse errors, the error baseline format is used inline for that file.
- If `declaration: true` and the `.d.ts` files compile with errors, a `//// [DtsFileErrors]`
  section is appended containing the error baseline of the declaration files.
- If `noCheck` is not set, the harness re-compiles with `noCheck: true` and diffs the output.
  Any differences appear as `!!!! File ... differs from original emit in noCheck emit` sections.
- If `noEmit: true` or `emitDeclarationOnly: true` and no JS is produced, the baseline is
  `null` (no `.js` baseline file is created).

### 1.2 `.errors.txt` — Error Diagnostic Baseline

**Contains:** Compiler diagnostic messages interleaved with source code, with error spans marked
by `~` squiggles.

**Generated by:** `Compiler.doErrorBaseline()` and `Compiler.getErrorBaseline()` in `harnessIO.ts`.

**Key behavior:** If there are zero diagnostics, the baseline is `null` — no `.errors.txt` file
is produced for tests that compile without errors.

**Structure:**
```
file.ts(line,col): error TSnnnn: message

==== file.ts (N errors) ====
    <source line>
    ~~~~~~~~~~~
!!! error TSnnnn: message text
!!! related TSnnnn file.ts:line:col: related message
    <next source line>
```

### 1.3 `.symbols` — Symbol Baseline

**Contains:** For each identifier/expression node in the AST, the resolved symbol information.

**Generated by:** `Compiler.doTypeAndSymbolBaseline()` using `TypeWriterWalker.getSymbols()`.

**Structure:**
```
//// [tests/cases/compiler/foo.ts] ////

=== foo.ts ===
class Cell {
>Cell : Symbol(Cell, Decl(foo.ts, 0, 0))
}
```

Each symbol annotation line starts with `>` and shows `sourceText : Symbol(name, Decl(file, line, char), ...)`.
Library file declarations use `--` for line and character: `Decl(lib.es5.d.ts, --, --)`.

### 1.4 `.types` — Type Baseline

**Contains:** For each expression/identifier node, the inferred or declared type.

**Generated by:** `Compiler.doTypeAndSymbolBaseline()` using `TypeWriterWalker.getTypes()`.

**Structure:**
```
//// [tests/cases/compiler/foo.ts] ////

=== foo.ts ===
class Cell {
>Cell : Cell
>     : ^^^^
}
```

Each type annotation has two lines:
1. `>sourceText : typeString` — the type of the expression
2. `>            : ^^^^` — underline markers showing which parts are synthetic (compiler-generated)
   vs parsed. The `^` characters mark synthetic portions of the type node.

### 1.5 `.sourcemap.txt` — Source Map Record

**Contains:** Detailed mapping of emitted positions back to source positions.

**Generated by:** `CompilerTest.verifySourceMapRecord()`. Only produced when `sourceMap`,
`inlineSourceMap`, or `declarationMap` options are set.

**Structure:**
```
===================================================================
JsFile: foo.js
mapUrl: foo.js.map
sourceRoot:
sources: foo.ts
===================================================================
-------------------------------------------------------------------
emittedFile:foo.js
sourceFile:foo.ts
-------------------------------------------------------------------
>>>emitted line content
1 >
2 >^^^^^^^
1 >source mapping
2 >Emitted(1, 8) Source(1, 8) + SourceIndex(0)
```

### 1.6 `.js.map` — Source Map JSON

**Contains:** The raw JSON source map files, each under a `//// [filename.js.map]` header.
Also includes a visualization link to `sokra.github.io/source-map-visualization`.

**Generated by:** `Compiler.doSourcemapBaseline()`. Only produced when `sourceMap` or
`declarationMap` is enabled (not `inlineSourceMap`).

### 1.7 `.trace.json` — Module Resolution Trace

**Contains:** JSON array of module resolution trace entries.

**Generated by:** `CompilerTest.verifyModuleResolution()`. Only produced when
`traceResolution: true` is set.

## Section 2: Baseline Comparison

### 2.1 File Locations

Baselines live in two directories:
- **Reference (expected):** `tests/baselines/reference/<filename>.<ext>`
- **Local (actual):** `tests/baselines/local/<filename>.<ext>`

The `Baseline.runBaseline()` function compares actual output against the reference file.

### 2.2 Comparison Algorithm

From `Baseline` namespace in `harnessIO.ts`:

1. **Generate actual output** — the test produces a string (or `null` for "no baseline needed").
2. **Read reference file** — if the reference file exists, read its content. If not, use
   `"<no content>"` as the expected value.
3. **Encode actual** — the actual string is passed through `Utils.encodeString()` before comparison.
4. **Compare** — if `expected !== encodedActual`:
   - If actual is `null`/`"<no content>"`, write a `.delete` marker file at
     `tests/baselines/local/<filename>.delete`.
   - Otherwise, write the actual output to `tests/baselines/local/<filename>`.
   - Throw an error: `"The baseline file <filename> has changed."` with instructions to run
     `hereby baseline-accept`.
5. **If equal** — test passes, no files written.

### 2.3 Normalization

The harness applies these normalizations before comparison:

- **Path prefixes:** `Utils.removeTestPathPrefixes()` strips absolute test directory paths,
  replacing them with relative paths.
- **Library file positions:** Line/column numbers in `lib.*.d.ts` references are replaced with
  `--` (e.g., `lib.es5.d.ts(--,--)`) to avoid brittle baselines that break when lib files change.
- **Newline:** The harness constant `harnessNewLine = "\r\n"` — all baselines use CRLF line
  endings.
- **Byte order marks:** `Utils.removeByteOrderMark()` strips BOMs from output files before
  collation.

### 2.4 Multi-file Baseline Comparison

`Baseline.runMultifileBaseline()` handles baselines split across multiple files (used for
`.errors.txt` where each source file gets its own baseline file):

1. Each yielded `[name, content, count]` tuple creates a separate baseline file at
   `<relativeFileBase>/<name><extension>`.
2. Files with `count === 0` are skipped (no errors in that file).
3. After writing, scans the reference directory for files that exist in reference but were not
   written — these are "missing" and get `.delete` marker files.
4. Collects all errors and reports them together.

### 2.5 Baseline Acceptance

Running `hereby baseline-accept` copies files from `tests/baselines/local/` to
`tests/baselines/reference/`, and processes `.delete` marker files by removing the corresponding
reference files.

## Section 3: Baseline Formatting

### 3.1 `.js` Baseline Format

The `.js` baseline is constructed by `doJsEmitBaseline()`:

```
//// [tests/cases/compiler/<testFileName>] ////\r\n
\r\n
//// [<sourceFile1BaseName>]\r\n
<source1 content>
//// [<sourceFile2BaseName>]\r\n
<source2 content>\r\n        ← extra \r\n between last source and first JS output
\r\n
//// [<jsFile1BaseName>]\r\n
<js1 content>
//// [<jsFile2BaseName>]\r\n
<js2 content>\r\n
\r\n
//// [<dtsFile1BaseName>]\r\n     ← only if declarations generated
<dts1 content>
```

**Details:**
- The header line is `//// [tests/cases/compiler/testName.ts] ////` (full relative path).
- Source file section headers use just the base filename: `//// [foo.ts]`.
- JS output file sections also use just the base filename: `//// [foo.js]`.
- Sources are listed with `otherFiles` first, then `toBeCompiled` files.
- Each source line uses its original content with a `\r\n` separator between files.
- The last source file does NOT get a trailing `\r\n` before the separator.
- The `fileOutput()` helper formats each output file as `//// [basename]\r\n` + file text.
- If `fullEmitPaths` harness setting is true, full paths are used instead of base names.

### 3.2 `.errors.txt` Baseline Format

Generated by `iterateErrorBaseline()`:

**Part 1: Diagnostic Summary**
```
<formatted diagnostics using ts.formatDiagnostics()>\r\n
\r\n
```

**Part 2: Global Errors** (errors not associated with any file)
```
!!! error TSnnnn: message
```

**Part 3: Per-file errors** (for each input file)
```
==== <filePath> (N errors) ====\r\n
    <source line 1>\r\n
    <source line 2>\r\n
    ~~~~~~~~~~~~~~~\r\n          ← squiggle under error span
!!! error TSnnnn: Error message\r\n
!!! related TSnnnn file.ts:line:col: Related info message\r\n
    <source line 3>\r\n
```

**Squiggle generation:**
- Each line is prefixed with 4 spaces (`    `).
- Error squiggles replace non-space characters with spaces up to the error start position,
  then use `~` characters for the error span length.
- Multi-line errors continue squiggles across lines.
- Error messages are prefixed with `!!! <category> TS<code>: ` where category is `error`,
  `warning`, or `suggestion`.
- Related information lines use `!!! related TS<code> <location>: <message>`.

**Diagnostics are sorted** using `ts.compareDiagnostics` before processing.

### 3.3 `.symbols` Baseline Format

Generated by `iterateBaseLine(isSymbolBaseline=true)`:

```
//// [tests/cases/compiler/<testFile>] ////\r\n
\r\n
=== <filename> ===\r\n
<source code line>\r\n
>identifier : Symbol(name, Decl(file, line, char))\r\n
<next source line>\r\n
```

**Symbol format:**
```
Symbol(<symbolName>, Decl(<file>, <line>, <char>), Decl(<file2>, <line2>, <char2>), ...)
```

- Up to 5 declarations are shown; beyond that: `... and N more`.
- Library file declarations use `--` for line and character numbers.
- Symbol results are cached on declaration nodes via `__symbolTestOutputCache`.
- Only expression nodes, identifiers, and declaration names produce symbol entries.

### 3.4 `.types` Baseline Format

Generated by `iterateBaseLine(isSymbolBaseline=false)`:

```
//// [tests/cases/compiler/<testFile>] ////\r\n
\r\n
=== <filename> ===\r\n
<source code line>\r\n
>sourceText : typeString\r\n
>            : ^^^^underline\r\n      ← synthetic node underline (if present)
```

**Type display rules:**
- Types are printed using `TypeFormatFlags.NoTruncation | AllowUniqueESSymbolType | GenerateNamesForShadowedTypeParams`.
- If the file has no error baseline AND the type is `any` (not from expected contexts like
  imports, labels, binding elements, etc.), the intrinsic name (e.g., `error`) is shown
  instead of `any`.
- For type aliases like `type T = ...`, if the type resolves to just `T`, the type is
  re-printed with `InTypeAlias` flag to show the expanded form.
- The underline row uses `^` for synthetic (compiler-generated) type node portions and
  spaces for parsed portions. Only shown when the underline is non-empty.
- Lines containing only `{`, `}`, or whitespace between type annotations get no extra blank
  line separator; other source lines get a blank line before them.

### 3.5 Performance Stats Section

For `.types` baselines only, if any performance counter exceeds its threshold, a stats section
is prepended after the header:

```
//// [testFile] ////\r\n
\r\n
=== Performance Stats ===
Type Count: 1,000
Instantiation count: 2,500
Symbol count: 50,000
\r\n
\r\n
=== file.ts ===
...
```

Thresholds: Strict subtype cache (1000), Subtype cache (1000), Identity cache (1000),
Assignability cache (1000), Type Count (1000), Instantiation count (1500), Symbol count (45000).
Values are rounded to human-readable logarithmic values (10, 25, 50, 100, 250, 500, ...).

## Section 4: Parameterized Tests

### 4.1 Overview

TypeScript supports parameterized compiler tests where a single test source file produces
multiple baselines — one for each combination of parameter values. This is driven by
multi-value `// @directive: value1, value2` comments in the test source.

### 4.2 Which Directives Support Variation

The `CompilerTest.varyBy` static field defines which compiler options trigger separate test
runs when they have multiple values. It includes:

1. **All options** from `ts.optionDeclarations` that:
   - Are not command-line-only (`!isCommandLineOnly`)
   - Have type `"boolean"` or a map type (`typeof option.type === "object"`)
   - Have at least one "affects" flag: `affectsProgramStructure`, `affectsEmit`,
     `affectsModuleResolution`, `affectsBindDiagnostics`, `affectsSemanticDiagnostics`,
     `affectsSourceFile`, `affectsDeclarationPath`, `affectsBuildInfo`

2. **Explicit additions:** `"noEmit"`, `"isolatedModules"`

This covers options like `target`, `module`, `moduleResolution`, `moduleDetection`, `jsx`,
`strict`, `noLib`, `declaration`, `sourceMap`, `allowImportingTsExtensions`, etc.

### 4.3 Value Splitting

The `splitVaryBySettingValue()` function parses the directive value string:

```typescript
// @target: ES5, ES2015       → ["es5", "es2015"]
// @module: *, -amd            → all module values except "amd"
// @strict: true, false        → ["true", "false"]
```

**Rules:**
- Values are comma-separated and trimmed.
- All values are lowercased.
- `*` means "all valid values for this option" (from the option's type map or boolean values).
- Values prefixed with `-` or `!` are excluded.
- Single values (no commas, no `*`) do NOT trigger variation — the test runs once with that
  single value.
- The option's type map provides deduplication: if two string aliases map to the same numeric
  value, only one variation is created.

### 4.4 Cartesian Product

When multiple directives have multiple values, the harness computes the **Cartesian product**
of all variations. The `computeFileBasedTestConfigurationVariations()` function recursively
generates all combinations:

```
// @target: ES5, ES2015
// @module: commonjs, esnext
```

Produces 4 configurations:
1. `{target: "es5", module: "commonjs"}`
2. `{target: "es5", module: "esnext"}`
3. `{target: "es2015", module: "commonjs"}`
4. `{target: "es2015", module: "esnext"}`

**Safety limit:** The total variation count must not exceed 25. If it does, the harness throws:
`"Provided test options exceeded the maximum number of variations"`.

### 4.5 Baseline Filename Construction

Parameterized baselines encode the configuration in the filename using parentheses.
The `CompilerTest` constructor builds `configuredName`:

```
<basename>(<key1>=<value1>,<key2>=<value2>)<extension>
```

**Rules:**
- Keys are sorted alphabetically and lowercased.
- Values are lowercased.
- Multiple key-value pairs are comma-separated (no spaces).
- The parenthesized config is inserted between the base name and extension.

**Examples:**
```
// Source: abstractPropertyBasics.ts
// Directive: @target: ES5, ES2015

Produces baselines:
  abstractPropertyBasics(target=es5).js
  abstractPropertyBasics(target=es5).errors.txt
  abstractPropertyBasics(target=es5).symbols
  abstractPropertyBasics(target=es5).types
  abstractPropertyBasics(target=es2015).js
  abstractPropertyBasics(target=es2015).symbols
  abstractPropertyBasics(target=es2015).types
```

```
// Source: allowImportingTypesDtsExtension.ts
// Directives: @allowImportingTsExtensions: true, false
//             @moduleResolution: classic, node16, nodenext

Produces baselines like:
  allowImportingTypesDtsExtension(allowimportingtsextensions=false,moduleresolution=classic).errors.txt
  allowImportingTypesDtsExtension(allowimportingtsextensions=true,moduleresolution=node16).errors.txt
  ...
```

```
// Source: arrayIterationLibES5TargetDifferent.ts
// Directives: @noLib: true, false
//             @target: es5, es2015, esnext

Produces 6 configurations (2 x 3):
  arrayIterationLibES5TargetDifferent(nolib=false,target=es2015).js
  arrayIterationLibES5TargetDifferent(nolib=false,target=es5).js
  arrayIterationLibES5TargetDifferent(nolib=false,target=esnext).js
  arrayIterationLibES5TargetDifferent(nolib=true,target=es2015).js
  arrayIterationLibES5TargetDifferent(nolib=true,target=es5).js
  arrayIterationLibES5TargetDifferent(nolib=true,target=esnext).js
```

### 4.6 Test Suite Organization

For each configuration, the `CompilerBaselineRunner.checkTestCodeOutput()` creates a separate
`describe()` block with the configuration description appended:

```
compiler tests for tests/cases/compiler/foo.ts (@module: commonjs, @target: es5)
```

Within each describe block, 6 `it()` tests run:
1. `Correct errors for <file>` — `.errors.txt` baseline
2. `Correct module resolution tracing for <file>` — `.trace.json` baseline (conditional)
3. `Correct sourcemap content for <file>` — `.sourcemap.txt` baseline (conditional)
4. `Correct JS output for <file>` — `.js` baseline
5. `Correct Sourcemap output for <file>` — `.js.map` baseline (conditional)
6. `Correct type/symbol baselines for <file>` — `.symbols` and `.types` baselines

## Section 5: Test Configuration Parsing

### 5.1 Directive Syntax

Test configuration is specified via special comments in the test source file:

```
// @optionName: value
```

The regex used is: `/^\/{2}\s*@(\w+)\s*:\s*([^\r\n]*)/gm`

This matches:
- Lines starting with `//`
- Followed by optional whitespace
- Then `@` and one or more word characters (the option name)
- Then `:` and optional whitespace
- Then the value (everything until end of line)

The option name capture is case-insensitive when looked up (lowercased for comparison).

### 5.2 Multi-file Test Syntax

Multi-file tests use the `// @Filename:` directive to split a single test file into multiple
virtual files:

```typescript
// @target: es2015
// @module: commonjs
// @Filename: moduleA.ts
export class Foo { }

// @Filename: main.ts
import { Foo } from "./moduleA";
new Foo();
```

**Parsing rules (from `TestCaseParser.makeUnitsFromTest()`):**
- Lines before the first `// @Filename:` are treated as global options (not file content).
- Each `// @Filename:` starts a new file unit.
- File-level `// @option:` directives within a file unit are recorded in that unit's
  `fileOptions`.
- Content between `// @Filename:` directives becomes the file's source text.
- Lines are joined with `\n`.

### 5.3 Supported Directives

Directives map to either standard TypeScript compiler options or harness-specific options.

**Harness-specific options** (defined in `harnessOptionDeclarations`):
| Directive | Type | Purpose |
|-----------|------|---------|
| `allowNonTsExtensions` | boolean | Allow non-.ts file extensions |
| `useCaseSensitiveFileNames` | boolean | Case sensitivity for file system |
| `baselineFile` | string | Override baseline filename |
| `fileName` | string | Override test filename |
| `noErrorTruncation` | boolean | Don't truncate error messages (always true in harness) |
| `suppressOutputPathCheck` | boolean | Skip output path validation |
| `noImplicitReferences` | boolean | Don't auto-include reference files |
| `currentDirectory` | string | Override working directory |
| `symlink` | string | Define symlink |
| `link` | string | Define symlink (alternative syntax: `// @link: target -> linkPath`) |
| `noTypesAndSymbols` | boolean | Skip `.types` and `.symbols` baselines |
| `fullEmitPaths` | boolean | Use full paths in `.js` baseline file headers |
| `noCheck` | boolean | Enable noCheck mode |
| `reportDiagnostics` | boolean | Enable error collection in transpile baselines |
| `captureSuggestions` | boolean | Include suggestion diagnostics in error baselines |

**Standard compiler options** include all of `ts.optionDeclarations`: `target`, `module`,
`moduleResolution`, `moduleDetection`, `jsx`, `strict`, `declaration`, `sourceMap`,
`outFile`, `outDir`, `rootDir`, `noEmit`, `emitDeclarationOnly`, `isolatedModules`,
`preserveConstEnums`, `removeComments`, `newLine`, `lib`, `types`, `paths`, and many more.

### 5.4 Option Value Parsing

Values are parsed based on the option's declared type:
- **`"boolean"`**: `"true"` (case-insensitive) becomes `true`, anything else becomes `false`.
- **`"string"`**: Used as-is.
- **`"number"`**: Parsed with `parseInt(value, 10)`.
- **`"list"` / `"listOrElement"`**: Parsed via `ts.parseListTypeOption()`.
- **Map types** (e.g., `target`, `module`): Parsed via `ts.parseCustomTypeOption()` using
  the option's type map (e.g., `"es5"` -> `ScriptTarget.ES5`).

### 5.5 tsconfig.json in Tests

If any file in the test has the name `tsconfig.json` or `jsconfig.json`, it is extracted from
the file list and parsed as a project configuration. The parsed `compilerOptions` are used
as the base options, and the file list from the tsconfig determines which files are "to be
compiled" vs "other files".

### 5.6 File Classification

Files are classified as either "to be compiled" (passed to the compiler as root files) or
"other files" (available on the virtual file system but not directly compiled):

- **With tsconfig:** Files listed in `tsconfig.fileNames` are "to be compiled"; others are
  "other files".
- **Without tsconfig, with references:** If the last file uses `require()` or
  `/// <reference path="..."/>`, only the last file is "to be compiled"; all others are
  "other files" available via references.
- **Without tsconfig, without references:** All files are "to be compiled" together.

### 5.7 Symlink Directives

The `// @link:` directive creates virtual file system symlinks:

```
// @link: /real/path -> /symlink/path
```

Parsed by `parseSymlinkFromTest()` using regex:
`/^\/{2}\s*@link\s*:\s*([^\r\n]*)\s*->\s*([^\r\n]*)/gm`

### 5.8 Default Compiler Options

The harness sets these defaults before applying test directives:
- `newLine: CarriageReturnLineFeed` (unless overridden)
- `noErrorTruncation: true`
- `skipDefaultLibCheck: true` (unless explicitly set to false)
- `noResolve: false` (when no explicit compiler options provided)
